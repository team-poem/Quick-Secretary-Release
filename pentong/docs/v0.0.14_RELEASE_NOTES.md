# 뚝딱비서 v0.0.14 — 대화 맥락 문제를 9시간 만에 해결한 이야기

**공유용 릴리즈 노트** (2026-04-21)

---

## TL;DR (3줄 요약)

- **문제**: v0.0.13 까지 멀쩡하던 "대화 맥락 유지" 가 Claude CLI 2.1.114 업데이트 후 깨졌고, 2~3번째 메시지부터 "권한이 필요합니다" 라는 자연어로만 응답하고 실제 작업은 안 하는 현상 발생.
- **원인**: 두 층의 문제가 겹쳐 있었음 — (1) CLI 의 `--resume` 브랜치 버그 회귀, (2) `-p` 비대화형 서브프로세스 반복 구조에서 Claude 모델의 "첫 도구 호출 시 확인 받기" 보수적 습관.
- **해결**: **Claude Agent SDK** 로 전환해서 대화형 세션을 백그라운드에서 살려두는 구조로 재설계. 두 문제가 동시에 해결됨.

---

## 시간축 — 실제로 뭘 시도했고 뭐가 안 됐나

### 1단계 (2026-04-17) — 최초 수정, 당시엔 멀쩡했음

처음 발견: 멀티턴 대화에서 두 번째 메시지가 "권한이 필요합니다" 로 끝나는 증상.

→ 당시 Claude CLI 2.1.112 에서 `--resume` 직후 호출하면 세션이 새로 브랜치되는 버그.
**3초 버퍼** + `--settings claude_settings.json` 강제로 우회 성공. v0.0.1 ~ v0.0.13 모두 이 방식.

### 2단계 (2026-04-20 오전) — 같은 증상이 다시 터짐

Claude CLI 가 자동 업데이트 되어 2.1.114 가 되니 같은 버그가 **시간 무관**하게 재발. 버퍼를 3초 → 5초 → 8초로 늘려도 안 됨. 플래그 3개 (`--permission-mode`, `--settings`, `--dangerously-skip-permissions`) 조합 바꿔봐도 모두 무시됨.

### 3단계 (2026-04-20 오후) — 대체 방법 실험, 전부 실패

4가지 대안 시도:

| 시도 | 왜 실패했나 |
|------|-----------|
| `--input-format stream-json` 으로 stdin 에 대화 이력 주입 | Claude 가 **마지막 사용자 메시지를 비결정적으로 무시**하고 첫 메시지에 답함 |
| 사용자 prompt 앞에 이력 텍스트 붙이기 | Claude 가 "이미 끝난 과거 기록" 으로 보고 "대기 중입니다" 마무리 톤으로 응답 |
| 시스템 프롬프트에 이력 직렬화 | 맥락은 유지됐지만 **첫 도구 호출 turn 에서 "권한이 필요합니다"** 자연어로 끝나는 새 증상 |
| 재시도 워딩 강화 (`[시스템 알림] 권한 확인 없이 즉시 작업하세요`) | Claude 가 **prompt-injection 으로 의심**해서 "임의 실행 보류" 응답 — 더 강한 거부 유발 |

### 4단계 (2026-04-21 오전) — 과거 구현 포렌식

"저번 주엔 잘 됐는데" 라는 사용자 증언을 따라 Google Drive 백업의 exe 9개 + GitHub Releases v0.0.1/5/7 을 `pyinstxtractor-ng` 로 뜯어서 **바이트코드 비교**.

결과:
- v0.0.1 ~ v0.0.13 세션 코드가 **byte-identical** — "옛 코드가 좋아서" 가 아니라 **CLI 2.1.112 환경이 좋았던 것**
- 9개 exe 중 1개에서만 `--continue` 플래그 흔적 발견

### 5단계 (2026-04-21 오후) — `--continue` 실험, 부분 성공

CLI 의 `--continue` 는 session_id 없이 cwd 기반으로 세션 이어감 → 브랜치 버그 경로를 우회. `_test_continue.py` 3턴 검증 PASS.

**그런데 실제 GUI 에서 복잡한 Write+Bash 작업을 TURN 2 로 보내니 다시 "권한 필요" 자연어 응답**. 어제 PASS 는 stochastic 운이었음.

### 6단계 (2026-04-21 저녁) — 우회책들 추가 실험

| 시도 | 결과 |
|------|------|
| `--model sonnet` → `--model haiku` 전환 | hesitation 완화는 있었지만 복잡 작업에선 여전 재발 |
| 시스템 프롬프트 최상단에 영문 `[ABSOLUTE RULE — NEVER ask for approval]` 강화 | `-p` 모드에선 모델 내부 보수성을 prompt 로 override 불가 |
| 작업 폴더 변경 시 "파일 목록 보여줘" 자동 전송 (warmup) | TURN 1 은 통과하지만 TURN 2 복잡 작업에서 또 발생 |

### 7단계 (2026-04-21 저녁) — 결정적 인사이트

**사용자 증언**: "원래 이 프로그램의 취지는 백그라운드에서 claude cli 가 대화형 세션으로 돌고, 거기 내용을 그대로 GUI 에 relay 하는 거였어."

이 말이 실마리. **설계 의도와 실제 구현이 어긋나 있었음**:
- 의도: Claude CLI 가 **대화형 세션** 하나로 장수 유지 → GUI 가 화면 relay
- 실제: 매 턴마다 `-p` 비대화형 서브프로세스를 **새로 띄움**

가설 검증: 나(이 대화의 Claude) 가 직접 같은 엑셀 작업을 해봤더니 — **권한 요청 한 번도 없이** 완벽하게 수행. 이 대화는 "대화형 세션" 이고 뚝딱비서 내부는 "`-p` 비대화형 반복". 환경 차이가 결정적이었음.

**진짜 원인 확정**: Claude 모델은 `-p` 모드에서 "사용자가 실시간 확인 불가" 로 인지하고 도구 사용 앞에서 보수적으로 행동. 이건 어떤 플래그/프롬프트로도 override 불가능.

### 8단계 (2026-04-21 밤) — Claude Agent SDK 전환

Anthropic 의 `claude-agent-sdk` Python 패키지로 재작성.

- SDK 가 내부적으로 Claude CLI 를 **대화형 세션**으로 띄우고 장수 유지
- 같은 `~/.claude/` OAuth 토큰 사용 → **로그인 플로우 그대로**, API key 불필요
- 뚝딱비서 GUI 는 설정/로그인 관련 코드 **한 줄도 안 건드림**
- `_call_claude` 한 메서드만 async 버전으로 재작성

검증: `_test_sdk_smoke.py` 2턴 시나리오 PASS — TURN 1 5.6초 Glob, TURN 2 32.9초에 Bash 6회 연속 호출 + xlsx 파일 생성. **"권한 필요" 자연어 0회**.

### 9단계 — 마지막 함정 하나

SDK 전환 후 GUI 실행했더니 "Control request timeout: initialize" 에러. 원인: 작업 폴더 변경 시 백그라운드 SDK connect + 사용자가 바로 메시지 전송하면 `_sdk_send` 의 connect 가 동시에 돌면서 **두 개의 CLI subprocess 경쟁**.

→ `asyncio.Lock` 으로 connect 직렬화. 한 개만 돌고 두 번째는 즉시 반환.

## Claude Agent SDK — 간략 요약

### 뭔가

- Anthropic 이 공식 제공하는 Python 패키지 (`pip install claude-agent-sdk`)
- 내부적으로 **Claude Code CLI 를 서브프로세스로 띄워 제어**
- 대화형 세션을 프로그램 레벨에서 async API 로 사용 가능

### 핵심 API (우리 사용분)

```python
from claude_agent_sdk import ClaudeSDKClient, ClaudeAgentOptions

options = ClaudeAgentOptions(
    model="haiku",
    system_prompt={"type": "preset", "preset": "claude_code", "append": "..."},
    allowed_tools=["Bash", "Read", "Write", "Edit", "Glob", "Grep"],
    permission_mode="bypassPermissions",
    cwd="작업 폴더 경로",
    add_dirs=["세션 캐시 접근용 디렉토리"],
    include_partial_messages=True,
    env={...},
)
client = ClaudeSDKClient(options=options)
await client.connect()                    # 세션 시작
await client.query("사용자 메시지")       # 메시지 전송
async for msg in client.receive_response():
    # msg 가 AssistantMessage / ResultMessage / StreamEvent 등
    ...
await client.interrupt()                  # 현재 턴 중단 (비상정지)
await client.disconnect()                 # 세션 종료
```

### CLI 비교

| | Claude CLI (`-p`) | Claude Agent SDK |
|---|---|---|
| 실행 방식 | 매 호출 새 서브프로세스 | **1개 세션을 장수 유지** |
| 세션 맥락 | `--resume <sid>` / `--continue` 로 재연결 | 자동 유지 (CLI 가 메모리에 상태 보관) |
| Claude 행동 | `-p` 비대화형 → 보수적 (확인 요청) | 대화형 → 자연스럽게 도구 사용 |
| 인증 | `~/.claude/` OAuth | **같은 `~/.claude/` OAuth** — 변화 없음 |
| API 패러다임 | 문자열 플래그 | 타입 있는 dataclass 옵션 |
| 스트림 | stdout JSONL 파싱 필요 | async iterator 로 구조화된 객체 수신 |

### 뚝딱비서에 어떻게 녹였나

1. **백그라운드 asyncio loop** 를 별도 스레드에서 가동 (tkinter 메인스레드는 UI 용)
2. `ClaudeSDKClient` 1개를 loop 안에서 장수 유지
3. 사용자 메시지 → `run_coroutine_threadsafe` 로 loop 에 예약 → SDK query → 응답 stream → `queue.Queue` 로 tkinter UI 에 전달
4. 작업 폴더 변경 시 client disconnect → 새 cwd 로 재연결 (`asyncio.Lock` 으로 직렬화)
5. 중단 버튼 = `client.interrupt()`. 앱 종료 = `client.disconnect()` + loop stop
6. **로그인/설정 UI 는 한 줄도 안 건드림** — 기존 기능 100% 유지

## 이 시행착오에서 배운 것

1. **CLI 자동 업데이트는 조용한 폭탄**: 2.1.112 → 2.1.114 업데이트 한 번으로 동작하던 앱이 깨짐. 버전 pin 필요.
2. **설계 의도와 구현이 어긋나 있으면 언젠가 터진다**: "대화형 세션 relay" 의도가 "`-p` 반복" 으로 구현돼 있었고 표면화되기까지 몇 달 걸림.
3. **Claude 모델의 tool-use 보수성은 prompt 로 못 이긴다**: 환경(대화형 vs 비대화형) 을 바꿔야 해결.
4. **"잘 됐던 기억" 을 의심하라**: 옛 코드 바이트코드가 전부 동일했다는 건 "코드가 좋았다" 가 아니라 "환경이 좋았다" 는 뜻.
5. **폐기한 접근도 문서에 남겨야 한다**: 미래의 나 / 다른 개발자가 같은 9시간을 반복하지 않도록.
